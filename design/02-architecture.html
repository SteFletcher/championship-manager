02 · System Architecture

Software Architecture Description (SAD)

1. Architectural Style

Champman is a layered, engine-first single-page application with a strict separation between pure game logic and imperative UI. The design rests on three principles:

  1. Engine purity. Every module under src/engine/ is a pure ES module with no DOM access and no framework. It runs identically in the browser and in Node, which is what makes the 213-test suite possible without a DOM shim.
  2. Single source of randomness. One seeded mulberry32 RNG instance threads through the entire career. Determinism is a first-class architectural property, not an afterthought.
  3. UI as a thin client. js/app.js only renders state produced by the engine and forwards user gestures back to engine methods. It owns no game rules.
Key trade-off

The purity boundary costs a little boilerplate (the engine returns plain data objects the UI must re-render from) but buys deterministic testability and a clean place to put the entire game's correctness. This is the single most important decision in the architecture.

4. Dependency Rules

  • The dependency graph is a DAG with rng.js as the root. No cycles.
  • game.js depends on every engine module; it is the only module that composes the subsystems.
  • match.js depends only on rng.js and team.js — it knows nothing about seasons, finances, or transfers.
  • app.js imports Game and display helpers (teamRatings, overallRating, ability, familiarity, FORMATIONS, INSTRUCTION_AXES); it never imports match.js directly.
  • The data modules (teams.js, squads1995.js) depend on rng.js, players.js, and team.js (detailed positions) only — so the world can be rebuilt deterministically without the full engine.

5. The Determinism Contract

Determinism is enforced by three rules that every engine author must obey:

Rule 1 — One RNG

All randomness flows through a createRng(seed) instance. Math.random() appears exactly once in the entire engine, in match.js, solely to generate a seed when none is supplied. The career's RNG lives on game.rng and its state is serialized with the save.

Rule 2 — Serializable state

The RNG exposes getState() / setState(). Game.serialize() captures rngState alongside all club/world state; Game.restore() reconstructs the instance and re-injects the RNG state, so a resumed career draws the same future random numbers as the original.

Rule 3 — No hidden mutation

The match engine never mutates the player objects it is given; per-match state lives in stat line objects keyed by player id. This keeps the squad objects stable across the sim and the UI.

7. Match Engine Internals

MatchSim is a stateful object exposing playMinute() (returns the events generated that minute), setTactics(), makeSub(), and finish(). Internally each minute:

  1. Runs AI tactics policy (chase / shut up shop) and AI auto-sub policy for both sides.
  2. Resolves the midfield battle to decide possession.
  3. Generates quiet build-up play (passes, tackles) that feed the live ratings.
  4. Resolves fouls and cards (yellow → second-yellow → straight red).
  5. Resolves injuries with weighted victim selection and forced auto-subs.
  6. Decides whether the attacking side fashions a chance, then resolves it (miss / block / save / goal).
  7. Decays condition for every player on the pitch.
  8. Pushes a timeline snapshot (for replay/debug) and checks for stage end.

The unit-strength model is deliberately headcount-aware: a formation packing a unit strengthens it with diminishing returns, so 5-3-2 genuinely defends and 4-3-3 genuinely attacks — but midfield headcount moves possession only gently, so it doesn't drown out defensive/attacking intent.

8. Persistence Model

Saves are a single JSON blob in localStorage['champman-save']. The shape is a flat snapshot of the Game instance minus the RNG object (whose state is stored separately as rngState).

{
  version: 6,
  managerName, clubName,
  rngState,                 // ← the determinism key
  clubs: [...],             // full world state
  freeAgents, seasonIndex, week, calendar, fixtures,
  results, cup, cupResults,
  seasonStats, monthly, inbox, pendingOffers, nextOfferId,
  board, tactics, history, youthSeq,
  sacked, reputation, jobOffers, pendingJobOffer,
  scouted, pendingScouts
}

Restore constructs a throwaway Game (to get the class brand needed for private methods) and then Object.assigns the saved fields over it before re-injecting the RNG state. A version mismatch throws, refusing to load incompatible saves.
