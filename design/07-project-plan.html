07 · Project Plan

Waterfall phases, work breakdown, milestones & schedule

Methodology note

This plan presents the work as a classical waterfall sequence — requirements, design, implementation, integration, verification, deployment — which is the form the user asked the existing codebase to be reverse-engineered into. In reality the codebase was built incrementally with an engine-first, test-driven loop; this document reconstructs the shape a up-front plan would have taken.

4. Sequencing & Dependencies

  • Phase 3 (engine) must complete before Phase 4 (career) — game.js composes the engine modules.
  • Phase 4 must complete before Phase 5 (UI) — the UI renders Game state.
  • The match-day UI (5.8–5.10) depends on the resumable MatchSim from Phase 3.
  • The browser smoke test (6.3) depends on the entire UI being live.
  • Statistical realism (6.1) can begin as soon as Phase 3 lands and runs in parallel with Phase 4.

5. Effort Profile

The bulk of complexity is concentrated in two modules — match.js (867 LOC) and game.js (1122 LOC) — which together are over half the non-test codebase. Phases 3 and 4 are therefore the longest; UI work in Phase 5 is substantial but lower-risk because it is a pure consumer of tested engine state.
