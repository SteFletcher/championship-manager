10 · Horizon 1 Feature Specifications

Full designs for the three near-term roadmap items: assistant takes charge, league statistics, and player interaction

1. Purpose & Scope

The Roadmap names three near-term (Horizon 1) features. This document takes each from a one-paragraph pitch to a buildable specification: requirements, engine design, data-model deltas, UI design, and test plan. Unlike documents 01–08, which describe what is, this document — like the roadmap — describes what comes next.

Design rules inherited from the roadmap

Engine first — every mechanic lands in src/engine/ with tests before its UI. Determinism — all randomness flows through this.rng; a delegated week or a squad talk must replay identically from the same seed. One save bump — the two features that persist new state (STA, INT) share a single SAVE_VERSION migration (§5.1). Update 2026-07-15: the Engine Evolution track has since taken v7 (EE-2), v8 (EE-3), and v9 (EE-4), so this spec's bump is now v9 → v10; the migration content is unchanged.

2. Feature AST — Assistant Takes Charge

Let the player resolve a match week — or several — without watching the match. The assistant picks the strongest available XI, uses the manager's saved tactics, and files a report in the inbox.

2.1 Requirements

IDRequirementPriority
AST-01The pre-match screen offers an Assistant takes charge action that resolves the user's fixture without entering the live match screen.Must
AST-02A delegated match uses the manager's saved formation and mentality (from game.tactics) with the strongest available XI auto-picked by selectXI — never the generic AI 4-4-2 / normal.Must
AST-03A Holiday action advances up to N weeks (N ∈ {2, 4, "to season end"}), delegating any user fixtures, and stops early when: the manager is sacked, the season ends, an incoming transfer offer arrives, or a poach offer from a bigger club arrives.Must
AST-04Every delegated match produces an Assistant's report news item: score, scorers, and a one-line verdict.Must
AST-05Delegation is deterministic: the same save + seed + delegation choices replays the identical career. No extra RNG draws compared to the existing auto-sim path.Must
AST-06Weekly upkeep (contracts, injuries healing, board, transfers, expansions, scouts) runs unchanged during delegated weeks — holiday skips watching, never simulating.Must
AST-07The week summary screen visually distinguishes a delegated user result (assistant icon/tag) from a live-played one.Should

2.2 Engine Design

The machinery mostly exists: advanceWeek(null) already auto-simulates the user's fixture via #resultFor → simulateMatch(aiSetup(home), aiSetup(away)) (game.js:281). Two gaps remain, and they define the work:

  1. aiSetup ignores the user's tactics. aiSetup() (game.js:195) hard-codes formation: '4-4-2', mentality: 'normal' for every club, including the user's. Fix: when clubName === this.clubName, read this.tactics.formation and this.tactics.mentality. The full-squad setup shape is kept so selectXI handles injuries and bans automatically (AST-02). This also quietly improves the existing sacked/no-show sim paths.
  2. The UI never offers the path. onContinue() (app.js:707) forces showPreMatch(fixture) whenever a user fixture exists. The engine needs no new simulation code — only a thin API for the UI to call.
// game.js — new public API
delegateWeek()
  // advanceWeek(null) + files the Assistant's report news item
  // for the user fixture (if any). Returns lastResults.

holiday({ weeks })            // weeks: number | 'season'
  // loop: delegateWeek(); stop when
  //   sacked || seasonEnded || pendingOffers grew || jobOffers grew || weeks done
  // Returns { weeksAdvanced, stoppedBy }  // 'done'|'sacked'|'offer'|'poach'|'seasonEnd'
Determinism note (AST-05)

The delegated path is byte-for-byte the existing auto-sim path — one rng.int seed draw per match, in fixture order. Changing aiSetup to read saved tactics changes no RNG call: formation and mentality only alter how the drawn seed's match plays out inside simulateMatch, which owns its own child RNG. The news item is written after simulation and draws nothing.

2.3 Data Model Changes

None persisted. holiday() runs synchronously to its stop condition, so no "on holiday" flag needs to survive a save/load. No SAVE_VERSION impact.

3. Feature STA — League Statistics Pages

A dedicated Stats screen with league-wide leaderboards: goals, assists, average rating, discipline, and clean sheets — per division. The roadmap calls this "pure UI"; in truth the aggregation needs two small engine additions first (discipline and clean sheets are not yet accumulated).

3.1 Requirements

IDRequirementPriority
STA-01A new Stats hub screen shows five leaderboards as tabs: Goals, Assists, Average Rating, Discipline, Clean Sheets.Must
STA-02Each board filters by division (Div 1 / Div 2), defaulting to the user's division, and shows the top 20.Must
STA-03seasonStats additionally accumulates yellows, reds, and cleanSheets per player (the per-match lines already carry cards; clean sheets are derivable at result time).Must
STA-04Average Rating requires a minimum of 6 appearances to qualify (prevents a one-match 9.0 topping the board).Must
STA-05A clean sheet is credited to every natural-GK player with minutes > 0 on a side that conceded 0 in that match.Must
STA-06Discipline ranks by reds desc, yellows desc; the board shows both columns.Must
STA-07Rows for the user's club are highlighted, matching the league-table convention (05b).Should
STA-08Players who left the league mid-season (sold to a club, still in-world) keep their stats; ids that no longer resolve to any club are skipped without error.Should
STA-09Boards reset with seasonStats at season end (existing behaviour); the season-review news keeps its golden-boot line.Must

3.2 Engine Design

seasonStats (game.js:371) is a flat map playerId → {apps, goals, assists, ratingSum, ratingN, motm} populated in #applyMatchResult for every club, not just the user's — so the league-wide data pipeline already exists. Three additions:

  1. Accumulate cards. In the per-line loop: stats.yellows += line.yellow; stats.reds += line.red; (the MatchResult lines already carry both — match.js:124).
  2. Accumulate clean sheets. Once per side per match: if conceded === 0, credit each appearing natural-GK per STA-05. #applyMatchResult already has conceded in scope.
  3. A selector so the UI stays dumb. 
    // game.js — new public API
leagueLeaders({ metric, division, limit = 20 })
  // metric: 'goals'|'assists'|'rating'|'discipline'|'cleanSheets'
  // → [{ id, name, pos, club, division, apps, value, value2? }]
  //   value2 = yellows on the discipline board
  // Resolves ids by scanning club rosters; skips unresolvable ids (STA-08);
  // applies the 6-app floor for 'rating' (STA-04). Pure read — no RNG.

4. Feature INT — Player Interaction & Morale Events

Give morale inputs and consequences. Today it is nearly inert: results move it (+8 W / +1 D / −6 L), listing (−10) and renewing (+8) nudge it, and its only output is a ±3% sharpness factor in the match engine (match.js:237: (morale − 70) × 0.001). This feature adds manager-driven talks, morale-driven events, and enough match-engine weight that the loop is worth playing.

4.1 Requirements

IDRequirementPriority
INT-01Three per-player talks from the Squad screen: Praise, Warn, Fine. Each returns a visible reaction (news-style line).Must
INT-02One talk per player per 3 weeks (TALK_COOLDOWN_WEEKS = 3); buttons disable with a tooltip during cooldown.Must
INT-03Praise devalues when hollow: half effect if morale ≥ 85, or if the player was already praised within the last 6 weeks.Must
INT-04Fines are justified only after a red card, a five-booking suspension, or a last-match rating < 6.0. An unjustified fine backfires (morale −12, "dressing-room leak" news). A fine credits one week's wage to the club balance.Must
INT-05Warning an out-of-form player is a gamble resolved via this.rng: 60% respond (form +0.4, morale −2), 40% sulk (morale −8).Must
INT-06Weekly morale events: a player with morale < 40 for 3 consecutive weeks submits a transfer request (news + wantsAway flag, fires once until resolved).Must
INT-07Responding to a transfer request: transfer-listing the player restores +8 morale (he sees a way out); refusing (dismiss) costs −6 and he stays wantsAway. Selling or a morale recovery ≥ 60 clears the flag.Must
INT-08wantsAway players are preferred targets in #maybeAiOfferForUserPlayer (offer chance ×2) and accept for-sale moves readily.Should
INT-09Playing-time unhappiness: from week 10, a player in the squad's top 8 by ability with 0 apps in the last 5 matchdays loses 2 morale/week ("wants first-team football" news, at most once a month).Should
INT-10Morale weight in the match engine doubles: sharpness term (morale − 70) × 0.002 (a ±5.8% swing), tuned to be measurable but smaller than form's ±8%.Must
INT-11All talk outcomes and event rolls draw from this.rng in a fixed order; replays are identical.Must

4.2 Engine Design

New public API on Game, following the toggleTransferList/renewContract pattern (mutate + clamp + news + return):

praisePlayer(id)  // → { ok, reaction }   morale +6 (half per INT-03), sets lastPraiseWeek
warnPlayer(id)    // → { ok, reaction }   INT-05 gamble; requires form < 6.0
finePlayer(id)    // → { ok, reaction }   INT-04; justified: morale −5, wage → balance
                  //                      unjustified: morale −12 + leak news
// all three: reject (ok:false) during cooldown; set lastTalkWeek = this.week

Weekly events run as a new #moraleEvents() step inside advanceWeek, after #weeklyUpkeep() and before the AI transfer step (so a fresh transfer request can influence the same week's AI bids, INT-08). Iteration is roster order over the user's club only — AI squads keep the cheap result-driven morale model.

#moraleEvents()  — user club, each player in roster order
  morale < 40 ?  unhappyWeeks++            : unhappyWeeks = 0
  unhappyWeeks ≥ 3 && !wantsAway ?
      wantsAway = true
      news("Transfer request", "…hands in a written request…")
  wantsAway && (sold || morale ≥ 60) ?  wantsAway = false
  week ≥ 10 && topEightAbility && appsInLast5 == 0 ?
      morale −2 ; monthly-capped "wants first-team football" news
Tuning target (INT-10)

Loss weeks cost −6 morale, so a relegation-battle squad trends toward the 20 floor. The talk economy must be a real counterweight: praising ~5 players a week (+6 each, cooldown-limited) roughly offsets one loss across the squad. The statistical test below pins the sharpness effect so tuning changes that break the counterweight fail CI rather than silently flattening the feature.

5. Cross-Cutting Concerns

5.1 Save Migration — SAVE_VERSION 9 → 10

(Originally specified as 6 → 7; renumbered after EE-2/EE-3/EE-4 took v7–v9.) STA and INT both persist new fields; they ship under a single version bump to avoid two-version churn. The migration is additive-only:

Payload areaChangev9 → v10 default
seasonStats[*]+ yellows, reds, cleanSheets0, 0, 0
clubs[*].players[*]+ lastTalkWeek, lastPraiseWeek, unhappyWeeks, wantsAwaynull, null, 0, false

Loading an older save fills defaults and stamps the new version (chaining through the existing v6→…→v9 migrations first); a one-time inbox note explains that card/clean-sheet counts start from the load point. No field is removed or reinterpreted, so new code never mis-reads old data.

5.2 Determinism Checklist

  • AST adds zero RNG draws — it reuses the existing auto-sim path.
  • STA is pure aggregation and pure reads — zero draws.
  • INT draws only in warnPlayer (one chance()) and #moraleEvents/#maybeAiOfferForUserPlayer adjustments, all via this.rng in fixed roster order. No Date, no Math.random.
  • The existing test "identical seeds replay the identical career" is extended to a script that praises, warns, fines, delegates, and holidays — the strongest single guard across all three features.

5.3 Delivery Order

BUILD 1 STA · league statistics

Smallest engine delta (two accumulators + one selector) and it forces the SAVE_VERSION 7 migration scaffolding that INT reuses. Ships value on day one.

BUILD 2 AST · assistant takes charge

No persistence work at all once the aiSetup tactics fix lands; mostly UI wiring on the Continue flow. Also the feature that makes long-career testing of INT faster (holiday through seasons).

BUILD 3 INT · interaction & morale

Largest surface and the only one with tuning risk (INT-10). Lands last, on top of the v7 migration from Build 1 and validated at speed with Build 2's holiday loop.

6. Acceptance Criteria (Horizon 1 exit)

Horizon 1 is done when
  1. A full season can be completed without watching a single match, using delegation and holiday, and replays identically from its seed.
  2. The Stats screen answers "who leads my division in goals, assists, rating, cards, and clean sheets" in two clicks, with the user's players highlighted.
  3. A player can be driven to a transfer request and talked back from one — and the morale difference is visible in results, not just in a column.
  4. The engine test suite grows by ~20 tests (§2.5, §3.5, §4.5) and stays green, including the extended determinism replay.
  5. v6 saves load cleanly into v7 with a single explanatory inbox note.
