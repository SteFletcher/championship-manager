11 · Engine Evolution Roadmap

Seven session-sized milestones that take the match engine from an abstract minute simulator to a watchable, player-level game — without ever breaking what works

Track status — 5 of 7 shipped

EE-1 (extensible core & regression armour), EE-2 (ten positions & formation geometry, save v7), EE-3 (eight real attributes, save v8), EE-4 (per-player instructions, save v9) and EE-5 (ball location & possession chains — the track's big re-calibration, landed with every invariant band green) are built, tested, and merged. The suite has grown from 148 to 226 tests. Next up: EE-6 · the player-level action stream.

The vision

Football Manager superseded Championship Manager by simulating the game, not just the result: every player passing, tackling, shooting, and making runs; a wide palette of positions; per-player tactical instructions set by clicking a player in the tactics view; and ultimately a match you can watch unfold. This roadmap evolves our engine toward that destination in seven independent, shippable steps. Each step lands with the game fully playable, the test suite green, and saves migrated.

2. Why a Track, Not a Rewrite

A big-bang engine rewrite was considered and rejected. The current engine is deterministic, statistically calibrated (scorelines land close to real football's near-Poisson distributions), and covered by 226 tests (148 at track start). A rewrite gambles all of that at once. Instead, each milestone changes one layer and re-proves the whole system before the next begins. The first milestone (EE-1) built the safety apparatus — a golden-master harness and a statistical invariant suite — that every later milestone leans on; EE-2 through EE-5 have already shipped through it with the invariants green throughout.

The momentum bar, done honestly

The roadmap rejected a momentum meter as "cosmetic; no mechanic attached". EE-5 revisits it with the mechanic attached: once the ball has a location and possession forms chains, spells of pressure exist in the simulation itself, and the CM-style territory bar simply displays true state. Calibration keeps the realism finding intact — shots cluster into spells; goals stay near-Poisson.

3. The Seven Milestones

EE-1 SHIPPED 2026-07-12

Refactor the simulator into a phase pipeline; pin its behaviour

Decomposed playMinute() into named, ordered phases sharing a minute context; made every event machine-readable (structured data payload, not just commentary text); and pinned behaviour with golden-master fixtures plus a statistical invariant suite. Byte-identical results before/after — proven, not promised. Ships with npm run golden:update regeneration tooling. Design →

EE-2 SHIPPED 2026-07-13 · save v7

From 4 units to 10 pitch positions with coordinates

GK, DR, DC, DL, DM, MR, MC, ML, AMC, ST. Formations became slot lists with pitch coordinates; the binary out-of-position penalty became a familiarity ladder (natural 1.0 · secondary 0.9 · same-unit 0.8 · cross-unit 0.65); squads and v6 saves migrate deterministically. Unit math is derived from position families, so the engine's existing balance carried over. Design →

EE-3 SHIPPED 2026-07-13 · save v8

From atk/def to 8 attributes the engine actually consults

Passing, finishing, tackling, positioning, pace, stamina (+ handling, reflexes for keepers). atk/def are now derived blends recomputed on every attribute write, so untouched code keeps working; each engine decision point (shot resolution, penalties, keeper duels, condition decay) reads the attribute that should drive it via attrOf(), with legacy fallbacks that reproduce the old formulas for plain atk/def players. Design →

EE-4 SHIPPED 2026-07-16 · save v9

Click a player on the tactics pitch, tell him what to do

The FM-defining interaction: forward runs / balanced / hold position, and press high / normal / sit deep — per slot, persisted with tactics, visualised as arrows on the pitch (with a team balance strip), and consumed by the engine as weight and strength modifiers. Defaults proven byte-identical to the pre-EE-4 goldens; statistical tests verify each axis moves play in the promised direction. AI managers derive presets from mentality on in-match switches. Design →

EE-5 SHIPPED 2026-07-17

The ball is somewhere, and it matters

The independent attacking-minute abstraction became a zone state machine (defensive third / middle / final third × possession) advanced in three ticks a minute, with a bounded territory-momentum term so pressure spells persist. Chances come only from the final third; territory, final-third entries, and a live CM-style territory bar display real state; the flow log feeds EE-7's camera. Calibrated back to the pre-EE-5 profile with all invariants green — shots now cluster into spells while goals stay near-Poisson. Design →

EE-6 ACT · player-level action stream

Named actions: who passed, who tackled, who ran

Each possession tick resolves into concrete actions — pass (passer → receiver), dribble, tackle, interception, cross, shot — chosen by attributes, position, and instructions. Stat lines stop being parallel bookkeeping and become sums over real actions. Commentary derives from the stream. Design →

EE-7 VIEW · the 2D match viewer

Watch the whole game unfold

An SVG pitch with 22 markers and a ball, animating the EE-6 action stream at selectable speed, with a highlights mode. A pure consumer of engine output — zero engine changes — which is the proof the extensibility programme worked. Design →

Build order: EE-1 ✅ → EE-2 ✅ → EE-3 ✅ → EE-4 ✅ → EE-5 ✅ → EE-6 → EE-7
  • EE-1 gates everything — no later milestone starts without the golden/statistical harness in place.
  • EE-2 before EE-4 and EE-7 — instructions attach to formation slots; the viewer places players at slot coordinates.
  • EE-3 before EE-6 — actions pick their probabilities from the specific attributes, not the old composites.
  • EE-5 before EE-6 — actions happen somewhere; the zone model supplies the where.
  • EE-2 ↔ EE-3 could swap; the chosen order puts the save-migration risk (positions touch selectXI, formations, and the tactics UI) earlier, while the squad is still simple.

6. Engineering Principles

The rules every milestone obeys
  1. Two-tier regression armour. Golden masters pin exact behaviour and are re-pinned deliberately (with a changelog note) when a milestone intends to change outcomes. Statistical invariants — scoreline distribution, home advantage, shots, cards, possession spread — must pass always, on every milestone, no exceptions. Realism is the contract; goldens are the tripwire.
  2. Additive schema, derived compatibility. New fields are added alongside old ones; old fields (pos, atk, def) become derived views so untouched code paths never notice. Nothing is removed until nothing reads it.
  3. One save bump per milestone, additive-only. Migrations fill defaults deterministically (seeded from the save's own seed where jitter is needed); a vN save always loads into vN+1.
  4. Determinism is non-negotiable. All randomness through the seeded RNG; presentation-layer randomness (viewer jitter) uses a separate RNG so watching a match can never change its result.
  5. Engine first, UI reads. Every mechanic lands in src/engine/ with tests before its UI, per the roadmap's standing rule.
  6. Session-sized or split. If a build session reveals a milestone is too big, the design's named "cut line" says what ships and what rolls into a follow-up session — never a half-landed engine change.

8. Relationship to the Feature Roadmap

This is an engine track, orthogonal to the feature horizons in 09 · Roadmap. Horizon 1 features (assistant, stats, interaction) touch game.js and the UI, not the match core — they can interleave freely between EE milestones. Two courtesy rules: coordinate SAVE_VERSION numbers at build time (EE-2 took v7, EE-3 v8, and EE-4 v9; the next milestone or Horizon feature that bumps the schema takes v10), and never run an EE milestone and a Horizon feature in the same session.

9. Acceptance (track exit)

The track is done when
  1. A full match can be watched on a 2D pitch — every pass, tackle, run, and shot attributable to a named player — or skimmed as highlights.
  2. Clicking any player in the tactics view sets individual instructions that measurably change how the team plays, verified statistically.
  3. Season-level outputs (league tables, scorelines, cards, injuries) remain inside the statistical invariant bands they satisfy today.
  4. A career save from before EE-2 loads cleanly at the end of the track.
  5. The same seed still replays the identical career, viewer on or off.
