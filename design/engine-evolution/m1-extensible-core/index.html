EE-1 · Extensible Core & Regression Armour

Refactor the match simulator into a phase pipeline, make every event machine-readable, and pin current behaviour so the rest of the track cannot silently break the game

1. Purpose

Every later milestone rewrites part of the minute loop. Today that loop is one 153-line method (playMinute(), match.js:478–631) plus a 84-line chance resolver, and the only protection around it is behavioural tests at the result level. This milestone changes no gameplay at all; it restructures the simulator so future changes are local, and builds the apparatus that proves — every session, automatically — that nothing else moved.

The contract of this milestone

Same seed, same inputs → byte-identical finish() output before and after the refactor. Goldens are captured from the current code first, the refactor lands second, and the goldens must still pass. This ordering is part of the spec, not an implementation detail.

2. Session Scope

In scopeOut of scope (deferred)
Golden-master harness + fixtures; statistical invariant suite; phase pipeline refactor of MatchSim; structured data on all events; regeneration tooling (npm run golden:update). Any behaviour change; any new mechanic; any UI change; save format change; zone/ball concepts (EE-5); event schema for actions that don't exist yet (EE-6 extends the same data field).

Cut line if the session runs long: the harness (goldens + invariants) ships even if only playMinute is decomposed and #resolveChance keeps its current shape — the armour matters more than the tidiness.

4. Engine Design

4.1 The phase pipeline

// match.js — the minute loop becomes data
static MINUTE_PHASES = [
  'halfTimeWhistle',   // minute-46 log line
  'aiPolicies',        // #aiTacticsPolicy + #autoSubPolicy, home then away
  'possession',        // midfield battle → ctx.attackerKey / ctx.defenderKey
  'buildUp',           // passes + tackle bookkeeping and ratings
  'discipline',        // fouls, yellows, reds — home then away
  'injuries',          // knocks + forced subs — home then away
  'chanceCreation',    // chance probability roll → #resolveChance
  'fatigue',           // condition decay + minutes
  'snapshot',          // timeline capture
  'clock',             // end-of-stage check
];

playMinute() {
  if (this.finished) return [];
  this.turnEvents = [];
  this.minute++;
  const ctx = { minute: this.minute, rng: this.rng };  // grows in later EEs
  for (const name of MatchSim.MINUTE_PHASES) this[`#phase_${name}`]?.(ctx) ?? this.#phases[name](ctx);
  return this.turnEvents;
}

Each phase is a private method extracted verbatim from today's loop — the refactor is a move, not a rewrite. possession writes ctx.attackerKey/ctx.defenderKey; buildUp and chanceCreation read them. The RNG threading is untouched (CORE-02), which is what makes byte-identity achievable.

4.2 Structured events

// #log gains a data parameter; text stays for display
#log(minute, type, sideKey, player, text, data = {})
// event shape (additive — existing fields unchanged):
{ minute, type, side, player, text,
  data: { playerId?, ...perType } }

// per-type payloads pinned in this milestone:
goal   → { playerId, assistId|null, score: {home, away} }
save   → { shooterId, keeperId }
miss / block / chance → { playerId }
corner → { }                    yellow/red/foul → { playerId }
injury → { playerId, weeks }    sub → { onId, offId }
tactics→ { formation, mentality }

Names stay in text; data carries ids so EE-6/EE-7 (and any future stats work) never parse strings. The timeline entries are already structured and are untouched.

4.3 Golden-master harness

test/golden/
  fixtures.json      // header: {generated, reason, engineNote}
                     // cases: [{name, seed, setupKind, knockout, sha256, score,
                     //          eventCount, summary}]
tools/update-goldens.js   // regenerates; prints old→new diff table
test/golden-master.test.js // replays each case, hashes canonical JSON, compares

Canonical serialization: JSON.stringify of finish() with sorted keys. The fixture stores the hash plus human-readable summary fields so a failure or regeneration diff is reviewable in the PR without decoding hashes.

4.4 Statistical invariants (the permanent contract)

~400 sims, two balanced mid-table squads, alternating venue. Bands are wide enough to absorb legitimate tuning, tight enough to catch a broken engine:

5. Data Model & Save Changes

None. No SAVE_VERSION impact. The event data field appears only in match results, which are not persisted beyond lastResults and are additive there.

6. UI Design

No player-facing change. The deliverable UI is developer-facing — what a golden failure looks like in the terminal, specified so failures are diagnosable at a glance:

$ npm test
✖ golden-master › seed 1189 · full-squads · league
    hash mismatch
    pinned:  score 2-1 · 47 events · a41f…09c2  (pinned 2026-07-12: EE-1 initial)
    current: score 2-1 · 48 events · 77d0…e1aa
    first divergence: minute 63 — extra event type "foul" (home)
    → intentional change? run: npm run golden:update -- --reason "..."

$ npm run golden:update -- --reason "EE-2 positional familiarity"
  seed 1189  2-1 → 2-1   47 → 48 events   re-pinned
  seed 2204  0-0 → 1-0   38 → 41 events   re-pinned
  8 fixtures updated · header reason recorded

The first-divergence line (replay both event streams, report the first differing index) turns a hash mismatch into a debuggable statement — worth the small extra tooling cost.

8. Exit Criteria

EE-1 is done when
  1. Goldens captured from pre-refactor code pass against the refactored engine.
  2. All 148 existing tests pass unmodified; suite grows by ~15.
  3. The statistical invariant suite exists, passes, and is documented as never-casually-rebaselined.
  4. A grep for text-parsing of events in js/app.js finds nothing new — the UI keeps consuming events as before.

9. Risks

RiskMitigation
Refactor subtly reorders an RNG drawGoldens catch it byte-for-byte; the first-divergence tool localises it to a minute and event.
Invariant bands flake (statistical noise)Fixed seed set for the 400 sims — the suite is deterministic, bands are checked against one known sample.
Phase decomposition tempts behaviour "fixes"Hard rule in scope: zero behaviour change; any bug found is filed, not fixed, this session.
