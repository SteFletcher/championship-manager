EE-2 · Positions & Formation Geometry

From four coarse units to ten pitch positions with familiarity grades and slot coordinates — the substrate for instructions, the viewer, and real squad-building depth

✅ Shipped 2026-07-13 · save v7

Built as designed (commit c863fe1): ten detailed positions with secondaries, formations as slot lists with pitch coordinates, the familiarity ladder in place of the binary out-of-position penalty, and deterministic v6→v7 save migration. Landed alongside the real 1995/96 squads, which pin detailed positions for real players.

1. Purpose

Today a player is a GK, DF, MF, or FW and is either in position or 25% worse (slotPenalty, team.js:26). FM's depth starts here: a right-back is not a centre-back, a winger is not a playmaker, and a formation is a shape on the pitch, not three numbers. This milestone introduces ten positions, formation slot maps with coordinates, and a familiarity ladder — while deriving the old units from position families so the engine's calibrated balance carries straight over.

2. Session Scope

In scopeOut of scope (deferred)
Position taxonomy + familiarity; formation slot maps with coordinates; selectXI slot assignment; match engine familiarity hook; squad generation + save migration; squad/tactics UI position labels; pitch view placement by coordinates. Per-player instructions (EE-4); attribute changes (EE-3); drag-and-drop slot editing in the tactics view (players are auto-assigned; manual slot swaps stay list-based as today); new formations beyond re-expressing the current five; secondary-position training.

Cut line: engine + data + migration ship with squad-screen labels; the pitch-view coordinate rendering can roll to the EE-4 session (which rebuilds that screen anyway).

4. Engine Design

4.1 Taxonomy & familiarity

// team.js
export const DETAILED_POSITIONS = ['GK','DR','DC','DL','DM','MR','MC','ML','AMC','ST'];
export const UNIT_OF = { GK:'GK', DR:'DF', DC:'DF', DL:'DF',
                         DM:'MF', MR:'MF', MC:'MF', ML:'MF', AMC:'MF', ST:'FW' };

export function familiarity(player, slotPos) {
  if (player.position === slotPos) return 1.0;
  if (player.secondaries?.includes(slotPos)) return 0.9;
  if (UNIT_OF[player.position] === UNIT_OF[slotPos]) return 0.8;
  if (player.position === 'GK' || slotPos === 'GK') return 0.4;
  return 0.65;                     // cross-unit outfield
}

The match engine's single integration point is #eff (match.js:278): slotPenalty(player, slot) → familiarity(player, entrySlotPosition). On-pitch entries carry the detailed slot; #unit/slotCounts group by UNIT_OF, so strengths, sendings-off maths, and formation switching keep their semantics.

4.3 Slot-based selectXI

Replace unit-count filling with per-slot assignment: order slots GK → DF → MF → FW; for each, pick the highest fitness(p) × familiarity(p, slot) unassigned player, requiring naturals-first per slot as today (keeps determinism and the existing "best XI" character). Bench selection unchanged. Complexity stays trivial (11 × squad size).

4.4 Generation & migration

// players.js — deterministic detailing (used by both generation and migration)
detailPosition(rng, unitPos, { atk, def }) →
  GK → GK
  DF → weighted pick: DC ×2, DR ×1, DL ×1
  MF → atk − def > 12 → AMC · def − atk > 12 → DM · else MR/MC/ML (MC ×2)
  FW → ST
secondaries: 35% of outfielders get 1 adjacent position (DC→DM, MR→MC, AMC→ST…)

Migration (SAVE_VERSION +1) runs detailPosition with an RNG seeded from the save seed + player id, so re-loading is idempotent and two loads agree.

8. Exit Criteria

EE-2 is done when
  1. A squad screen shows DR/DC/AMC/ST labels and the tactics pitch matches the formation's true geometry.
  2. Playing a natural MC at DM shows the amber tag and measurably weaker contribution (statistical test).
  3. An old save loads, migrates idempotently, and completes a match week.
  4. Invariants green; goldens re-pinned exactly once.
