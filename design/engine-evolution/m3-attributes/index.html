EE-3 · Attribute Expansion

From two composite numbers to eight attributes the engine actually consults — with atk/def surviving as derived blends so nothing untouched breaks

1. Purpose

FM's tactical depth rests on attributes that mean something distinct: a fast finisher plays differently from a strong passer at the same overall level. Today every engine decision reduces to atk, def, or their average. This milestone introduces six outfield attributes and two goalkeeping attributes, keeps atk/def as derived composites (so the UI tables, transfer valuation, scouting ranges, and any unconverted engine path keep working), and rewires each match-engine decision point to the attribute that should drive it. EE-6's per-action model then has real inputs to consume.

2. Session Scope

In scopeOut of scope (deferred)
Attribute schema + generation; derived atk/def; engine decision-point rewiring (§4.3); save migration; player profile attribute panel; scouting ranges per attribute; development/aging applied per-attribute. New mechanics (no new event types); weekly training focus (Horizon 2 item); hidden personality attributes beyond existing consistency / injuryProne; UI redesign of squad tables (they keep showing ATK/DEF composites this session — the profile shows the detail).

Cut line: schema, generation, migration, and derived composites ship with at minimum the three highest-value rewirings (pass completion, shot resolution, tackle success); remaining decision points (§4.3 rows 4–7) can follow next session without schema churn.

3. Requirements

IDRequirementPriority
ATTR-01Outfield attributes (1–99): passing, finishing, tackling, positioning, pace, stamina. Goalkeepers additionally: handling, reflexes (outfielders get low GK attrs, 1–15).Must
ATTR-02atk and def become pure derived getters: atk = .35·finishing + .30·passing + .20·pace + .15·positioning; def = .40·tackling + .35·positioning + .25·pace; GK def = .5·handling + .5·reflexes. Rounded to int, still 1–99.Must
ATTR-03Generation produces attribute profiles by position archetype (§4.2): a ST's finishing ≫ tackling, a DM's inverse; within-archetype spread via the squad-generation RNG.Must
ATTR-04Save migration expands each player's atk/def into the eight attributes deterministically (seeded from save seed + player id) such that the derived composites round-trip to within ±2 of the original values.Must
ATTR-05Engine decision points consume specific attributes per the wiring table (§4.3); no decision point consults raw atk/def directly in match.js when a specific attribute exists for it.Must
ATTR-06Condition decay scales with stamina: per-minute decay ×(1.25 − stamina/198), so a 90-stamina player fades visibly slower than a 40-stamina one.Must
ATTR-07Yearly development/aging (players.js) mutates individual attributes, not composites; pace and stamina decline first with age, positioning grows latest — matching football reality.Must
ATTR-08Scouting masks each attribute independently with the existing estimate-range mechanic; a scout report reveals all eight.Must
ATTR-09Player profile screen shows the eight attributes as labelled bars; squad/transfer tables keep composite ATK/DEF columns.Must
ATTR-10Statistical invariants hold; goldens re-pinned once ("EE-3 attribute wiring"). Transfer valuation (ability-based) shifts by < 5% mean across the league after migration.Must

4. Engine Design

4.1 Schema & compatibility

// players.js
export const OUTFIELD_ATTRS = ['passing','finishing','tackling','positioning','pace','stamina'];
export const GK_ATTRS = ['handling','reflexes'];

// Derived composites — defined once, used by ability(), UI, transfers, scouts:
export function atkOf(p) { /* ATTR-02 blend */ }
export function defOf(p) { /* ATTR-02 blend; GK branch */ }
// Plain data players keep literal atk/def in old saves until migration rewrites
// them; post-migration, atk/def are written-through on any attribute change.

Implementation choice: store computed atk/def as plain fields recomputed on any attribute write (not JS getters) — players are plain JSON across saves and structuredClone paths, and validateTeam keeps validating them untouched.

Emergent, intended consequences: a high-positioning defence concedes fewer clear chances even with modest tackling; pace matters at both ends (blend weights keep it secondary until EE-6 gives it runs and breaks to drive).

8. Exit Criteria

EE-3 is done when
  1. The profile screen shows eight attributes; scout reports range each one.
  2. The engine consults finishing/passing/tackling/handling at the wired decision points, proven by the statistical tests above.
  3. An old save migrates with composites within ±2 and plays on.
  4. Invariants green; one deliberate golden re-pin.
