EE-5 · Ball Location & Possession Chains

The ball is somewhere on the pitch and possession flows through zones — pressure spells become real simulation state, and the territory bar displays the truth

1. Purpose

Today each minute independently asks "who attacks?" and "is there a chance?" — there is no ball, no territory, no build-up between thirds. This milestone gives the simulation a ball state: which third it is in and who has it. Minutes become short chains of zone transitions; chances arise only from the final third; winning the midfield means living in the opponent's half rather than just winning a coin-flip more often. This is the platform EE-6 (actions happen in a zone) and EE-7 (the viewer shows the ball moving) stand on — and it resurrects CM's momentum bar with a mechanic attached.

Realism stance (carried from the roadmap discussion)

Real football's goals are near-independent (Poisson-like), but its shots and territory cluster into spells. The zone model must reproduce exactly that split: shot clustering up, scoreline distribution unchanged. The EE-1 invariant bands — especially goals/match, 0-0 rate, and the ≥5-goal cap — are the hard gate. Conversion probabilities stay independent per shot; no "hot streak" multiplier on finishing, ever.

4. Engine Design

4.1 The state machine

Zones from HOME's perspective (away's A is home's D):

  [home D] ◄──── [ M ] ────► [home A]
     ▲              ▲             ▲
   away attacks   contest      home attacks

Per tick, owner rolls (weights before normalisation):
  advance  = base 0.34 × zoneStep × instrMod
  hold     = base 0.38
  turnover = base 0.28 × pressMod × zonePressure

zoneStep:  M→A uses attack-vs-defence ratio  (2·atk/(atk+def), clamped .6–1.6)
           D→M uses midfield ratio (same shape)
turnover in own D is dangerous: opponent's next-tick chance gate is already open

// match.js — replaces possession/buildUp/chanceCreation phases
#phaseBallFlow(ctx) {
  for (let t = 0; t < TICKS_PER_MINUTE; t++) {          // 3
    const roll = rng.weightedPick(this.#tickWeights()); // advance|hold|turnover
    this.#applyTick(roll);                              // move zone / flip owner
    this.#flowLog.push({minute: ctx.minute, tick: t, ...this.ball});
    if (this.#inFinalThird(this.ball.owner)) {
      if (rng.chance(this.#tickChanceProb())) {          // ≈ BASE/3 rescaled
        this.#resolveChance(...);                        // unchanged resolver
        this.#resetAfterChance();                        // save→keeper D, goal→M
      }
    }
    this.#tickBuildUp();                                 // passes/tackles bookkeeping,
  }                                                      // zone-weighted, replaces old loop
}

Pass/tackle bookkeeping (ratings, stat lines) moves inside the tick with the same expected volumes as today (build-up passes ∝ held ticks). RNG draw count per minute changes — goldens are re-pinned once, invariants are the safety net, per the EE-1 doctrine.

4.2 Why pressure spells emerge (and stay honest)

A side that keeps winning advance rolls parks the ball in its attacking third; each held tick there re-opens the chance gate. That is shot clustering — from state, not from a bonus. Conversion per shot is untouched, so goals stay near-Poisson (BALL-08 asserts both halves of this). The 10-minute rolling territory bar therefore displays pressure rather than inventing it.

5. Data Model & Save Changes

No SAVE_VERSION bump. Ball state and flow log are per-match, in-memory, surfaced in the match result object (additive fields: stats.*.territoryPct, stats.*.finalThirdEntries, flow). lastResults consumers ignore unknown fields.

8. Exit Criteria

EE-5 is done when
  1. Watching a match, the territory bar visibly swings with spells of pressure, and the post-match stats corroborate it.
  2. The clustering split test passes: shots cluster, goals don't.
  3. All invariant bands pass; season-length sim produces a sane league table.
  4. The flow log exists in match results — EE-7's camera has its data source.
