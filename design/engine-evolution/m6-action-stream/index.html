EE-6 · Player-Level Action Stream

Every tick resolves into named actions — who passed to whom, who tackled, who ran — and the match record becomes a complete, replayable story

1. Purpose

EE-5 gave the match a ball; this milestone gives it protagonists. Each possession tick resolves into one or two concrete actions with a named actor, optional target, zone, and outcome. The stat lines (passes, tackles) stop being parallel bookkeeping and become sums over actions that actually happened. Commentary derives from the stream. After this milestone the engine produces everything a watchable match needs — EE-7 only has to draw it.

2. Session Scope

In scopeOut of scope (deferred)
Action vocabulary + resolver; actor/target selection from position, attributes, instructions; stat lines derived from actions; off-ball run actions for viewer choreography; commentary re-templated over actions; action log in match result. Any change to aggregate probabilities (shots, goals, turnovers keep EE-5's calibrated rates — actions decorate the tick outcome, they don't re-decide it); lateral channels; dueling animations; per-action xG; action- level ratings overhaul (rating deltas keep existing coefficients, now triggered per action).

Cut line: pass/tackle/shot/turnover actions with derived stats ship first; off-ball runs and enriched commentary templates are the tail and can roll over without schema change.

3. Requirements

IDRequirementPriority
ACT-01Action vocabulary: pass, throughBall, dribble, cross, tackle, interception, clearance, shot, save, block, claim, run. Each logged as {minute, tick, type, side, actorId, targetId?, zone, outcome} in an ordered action log.Must
ACT-02The tick outcome decided by EE-5 (advance/hold/turnover/chance) is narrated by actions, never re-decided: e.g. turnover → a defender's tackle (success) or a misplaced pass + interception, chosen by a single decorated roll.Must
ACT-03Actor selection: possession actions weight by position-zone plausibility (defenders start moves in D, midfielders in M, forwards in A — matrix in §4.2), familiarity, and instructions (forward-runners appear upfield more). Receiver ≠ passer, weighted toward the advance direction.Must
ACT-04Action outcome quality consults the specific attribute (pass → passing, dribble → pace, tackle → tackling, cross → passing/pace blend, shot chain unchanged from EE-3 wiring).Must
ACT-05Stat lines (passes, tackles, shots, saves…) are computed only from the action log — the old parallel counters are deleted; totals stay within today's realistic volumes (passes 250–500/side, tackles 10–30/side).Must
ACT-06Off-ball run actions (forward-runners during owned attacking-third ticks) exist for viewer choreography; zero probability impact this milestone; capped at 1/tick.Must
ACT-07Commentary is generated from actions via the existing template mechanism, upgraded to name both actors ("Reid slides in on Okafor", "Hurst → Vance, through ball!"); the events feed (goals, cards…) is unchanged in shape.Must
ACT-08Action log lives on the match result (result.actions), is not persisted to localStorage, and is bounded (~8–12 actions/minute ≈ ≤ 1,200/match).Must
ACT-09Live ratings react per action with the existing coefficient table (RATING); a match's rating spread stays within current norms (mean ≈ 6.6–7.0, no systematic inflation).Must
ACT-10Invariants hold; goldens re-pinned once ("EE-6 action narration"). Determinism: identical seed → identical action log.Must

4. Engine Design

4.1 Decoration, not decision

EE-5 tick outcome          EE-6 narration (one decorated roll each)
─────────────────          ──────────────────────────────────────────
advance                 →  pass(actor→receiver, fwd) | throughBall | dribble | cross(from wide MR/ML)
hold                    →  pass(lateral/back) — completion from passing attr
turnover                →  tackle(defender) | pass(actor, outcome:astray)+interception(defender) | clearance(in D)
chance → resolveChance  →  shot chain as today, logged as shot/save/block/goal actions

The aggregate rates stay EE-5's; the decorated roll only chooses which flavour of the already-decided outcome occurred and who did it. This is the design's load-bearing idea: EE-6 cannot unbalance the sim, because it never touches the probabilities that balance it.

4.2 Actor plausibility matrix (possession side)

ZoneDF weightMF weightFW weight
Own D3.0 (+GK claim/clearance)1.50.2
M1.03.01.0
Opp A0.3 (+forward-runners 1.0)2.03.0

Within a unit, weight × familiarity × sharpness as in #pick today. Defending-side actors (tackle/interception) mirror the matrix. The forward-runner bonus is EE-4's instruction surfacing in the narration — a marauding full-back now visibly appears in the final third.

4.3 Stat derivation replaces bookkeeping

// The tick build-up loop (#tickBuildUp) is deleted. Stat lines become:
onAction(a) {
  line(a.actorId).count(a.type, a.outcome);   // passes/tackles/shots/saves…
  rate(a);                                     // existing RATING coefficients
}
// finish(): line totals now provably equal action-log sums (ACT-05 test)

Volume calibration: 3 ticks/min × ~2.6 avg actions/tick ≈ 700 actions; passes land in the 250–450/side band. A constant ACTIONS_PER_TICK table keeps this tunable in one place.

5. Data Model & Save Changes

No SAVE_VERSION bump. result.actions is additive and unpersisted. Player stat-line shape is unchanged (same keys, new derivation). One compatibility note: result.events (goals, cards, commentary moments) keeps its exact shape — the UI feed and news generation read on untouched.

8. Exit Criteria

EE-6 is done when
  1. A match result contains a complete, deterministic action log from which every stat line is derived — verified by the equality test.
  2. Commentary names passer and receiver, tackler and victim.
  3. Aggregate match statistics are indistinguishable from EE-5's (invariants + volume bands).
  4. An EE-7 prototype question — "could a viewer replay this log?" — answers yes on inspection: every action has actor, zone, and outcome.

9. Risks
