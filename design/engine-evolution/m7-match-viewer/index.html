EE-7 · The 2D Match Viewer

Watch the whole game unfold: an SVG pitch replaying the action stream — full match or highlights — with zero engine changes

1. Purpose

The destination of the track. Twenty-two markers and a ball animate the EE-6 action log on a top-down pitch: passes travel, runs are made, tackles fly in, goals erupt. The viewer is a pure consumer of engine output — it changes no probability, draws nothing from the match RNG, and can replay any finished match from its result object. Shipping it without touching src/engine/ is the proof that the extensibility programme (EE-1…EE-6) delivered.

2. Session Scope

In scopeOut of scope (deferred)
SVG pitch component; marker layout from formation slot coordinates (EE-2) + zone elasticity (flow log); ball animation over actions; play/pause/speed (1×/2×/4×); highlights mode; integration into live match screen as a Pitch tab; replay of finished matches from the week summary. Sprite/kit graphics (markers are coloured discs + short names); off-ball AI choreography beyond EE-6 run actions and zone elasticity; camera zoom/follow modes; shareable replay URLs (Horizon 3 item); sound.

Cut line: highlights mode is the minimum shippable viewer (it needs only key-action sequences); full-match continuous mode is the stretch half. Build order runs highlights first for exactly this reason.

4. Design

4.1 Architecture: a replay machine, not a renderer of the sim

js/viewer/
  pitch.js       // SVG scaffold: markings, markers, ball, overlays
  layout.js      // (slots, flow, actions) → per-frame target coordinates
  timeline.js    // action log → keyframe schedule (full & highlight modes)
  animator.js    // rAF loop: eases markers/ball toward targets; speed control
// Input: { result, liveSim? }   — works from a finished result (replay)
//                                 or a paused/advancing MatchSim (live)

Live mode: the app already advances MatchSim.playMinute() on a timer; the viewer consumes that minute's actions and animates them across the minute's wall-time budget. Replay mode: same pipeline fed from the stored log. One code path, two sources — the EE-1 promise (structured data end-to-end) cashed in.

4.2 Coordinate synthesis

coordOf(action) =
  base   = slotXY(action.actorSlot, side)            // EE-2 geometry
  anchor = zoneShift(action.zone)                    // D/M/A band centre
  jitter = presRng(seed, minute, tick) × 6units      // deterministic sparkle
→ eased toward by animator; ball travels actor→target coords

This is deliberately theatre, not simulation — the engine knows zones, the viewer invents plausible metres. The design accepts that honestly (FM's own early 2D was the same trick), and VIEW-04 guarantees the theatre can never leak back into results.

5. Data Model & Save Changes

None. No SAVE_VERSION bump, no engine diff. If session review finds any engine change necessary, that is a design failure to record in the risk log — the correct fix lands as an EE-6 follow-up, not a viewer hack.

8. Exit Criteria

EE-7 — and the track — is done when
  1. A league match can be watched end-to-end on the pitch view: passes travel between named players, runs happen, tackles turn the ball over, goals go in.
  2. Highlights mode replays any past result from the week summary.
  3. git diff src/engine/ for the milestone is empty.
  4. The full test suite — goldens unre-pinned, invariants, smoke — is green.
  5. Track acceptance (11 · §9) reviewed and signed off against the shipped product.

9. Risks

RiskMitigation
Animation reads as chaos (markers teleporting)Zone elasticity + easing caps marker speed; jitter bounded to 6 units; highlight mode is the fallback watching experience.
Viewer tempts engine tweaks ("just expose one more field")Hard rule + golden no-re-pin test; needed fields become an EE-6 follow-up ticket.
Wall-time pacing vs sim minutes driftsAnimator owns the clock; sim minutes are consumed from a queue, never awaited on render.
Scope magnet (kits, zoom, sharing)Out-of-scope table is explicit; replay sharing already parked in Horizon 3.
